# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with
code in this repository.

This is a personal GNU Emacs configuration (`~/.emacs.d`), maintained since
2008. It is a configuration repo, not an application — there is no test suite.
"Running" it means loading the Emacs Lisp into a running Emacs.

## Layout & load model

`init.el` is the entry point. It runs `(server-start)`, extends `load-path`
with the source directories, then `require`s each config module **in a
deliberate, dependency-ordered sequence**. Order matters:

1. `basic-*` modules first — these have **no third-party dependencies**.
2. `straight-conf` (bootstraps the package manager), then dependent modules.
3. `custom/custom.el` is loaded as `custom-file` (auto-generated by Emacs
   Custom — do not hand-edit; it holds `custom-set-variables` /
   `custom-set-faces`).
4. `custom/langs/*` language modules last.

Source directories (all added to `load-path`):

- `custom/` — personal configuration modules, one concern per file
  (`*-conf.el`).
- `custom/langs/` — per-language setup (`java-conf.el`, `clojure-conf.el`,
  `go-conf.el`, …).
- `artifacts/` — vendored third-party `.el` files that are unavailable/misplaced
  in ELPA or carry small local patches. Kept in-repo intentionally.
- `solutions/` — reusable standalone Elisp libraries (e.g. `functions.el` = misc
  helpers).
- `foreign/` — a Go-based Emacs dynamic module (see below).
- `custom/mysnippets/` — YASnippet snippets, organized by major mode.

## Package management

**straight.el is the primary package manager** (bootstrapped in
`custom/straight-conf.el`, which fetches it over the network on first run).
Packages are declared with `straight-use-package`, often combined with
`use-package` (`:straight ...`). Language modules typically pin packages to a
specific git repo/branch:

```elisp
(straight-use-package
 '(cider :type git :host github :repo "clojure-emacs/cider" :branch "master"))
```

A legacy `package.el`/ELPA path also exists (`custom/elpa-conf.el` defines the
`use-elpa` helper), but new packages should use straight. The
`package-selected-packages` list inside `custom.el` is a legacy artifact, not
the source of truth.

## Conventions

- **Every module file ends with `(provide 'feature-name)`** and is wired into
  `init.el`'s require list. Adding a new module = create the file with a
  matching `(provide ...)`, then add a `(require ...)` to `init.el` in the
  correct position.
- **Personal interactive commands are namespaced with a `k/` prefix** (e.g.
  `k/clojure-eval-last-sexp`). Follow this when adding user-facing commands.
- Some config modules define a `*-highlight-initialize` function that adds
  `font-lock` keywords so custom macros (like `straight-use-package`) are
  highlighted in `emacs-lisp-mode`.

## `.gitignore` — whitelist model

`.gitignore` **ignores everything (`*`) by default**, then re-includes only
specific extensions (`*.el`, `*.org`, `*.md`, `*.go`, `*.yasnippet`) inside
explicitly whitelisted directories. A new file is tracked **only** if both its
extension and its directory are whitelisted — adding a file in a new directory
requires a new `!dir` entry. `communications.el`, `ejc-databases.el`, and
`.mc-lists.el` are explicitly ignored (local/secret data).

## Applying changes

There is no build step for the Elisp itself. To apply a change to a running
Emacs, eval the buffer/defun (`C-M-x`, `C-x C-e`) or `M-x load-file`; a full
restart re-runs `init.el`. Byte compilation is optional:
`emacs -batch -f batch-byte-compile custom/<module>.el`.

## `foreign/` — Go dynamic module

`foreign/` builds a C-shared Emacs module (`foreign-fmt.so`) from Go for fast
text manipulation (JSON/XML/EDN pretty-printing, in-file find/replace). Build
with `make` (or `make all`) inside `foreign/`; `foreign.el` auto-builds it on
load via `foreign-ensure-so`. It is currently **disabled** in `init.el` (the
`(require 'foreign)` line is commented out).

## Tooling notes

- **Tree-sitter** grammars and `*-ts-mode` mappings are configured in
  `custom/tree-sitter-conf.el`; install grammars via
  `M-x treesit-install-language-grammar`.
- **LLM integration** (`custom/llm-conf.el`) uses `gptel` pointed at a **local
  llama.cpp server** (OpenAI-compatible API on `localhost:8080`); the file
  header documents the `llama-server` invocation.
